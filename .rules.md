## Objetivo

Mantener consistencia técnica y funcional en `Matriz`, respetando lo ya implementado
y asegurando que cada cambio quede listo para despliegue en servidor Ubuntu con Docker Compose.

## Stack real del proyecto

- **Backend**: Python · FastAPI · SQLModel · SQLite (`sqlite:///./data/matrix.db`) · python-jose · bcrypt
- **Frontend**: React 19 · TypeScript · Vite 7 · Tailwind CSS 3 · shadcn/ui · Zustand · React Router v7 · Axios · Sonner · Recharts · Framer Motion
- **Auth**: JWT (HS256) con roles jerárquicos: `usuario → coordinador → admin → superadmin`
- **Infraestructura**: Docker Compose (`matrix_backend` + `matrix_frontend`) · Nginx en frontend · SQLite persistido en volumen `sqlite_data`
- **Módulos activos**: `auth`, `projects`, `matrix`, `dashboard`, `settings`, `roi`, `comments`, `drafts`, `webhook`, WebSocket (`/ws`)

## Estructura de carpetas — respetar siempre

```
backend/app/
  routes/       ← transporte HTTP: validación entrada/salida únicamente
  services/     ← lógica de negocio
  models/       ← modelos SQLModel (tabla)
  schemas/      ← Pydantic schemas de entrada/salida
  core/         ← seguridad, dependencias, WebSocket manager
  seeds/        ← datos iniciales (superadmin, preguntas matriz)
  config.py     ← Settings con pydantic-settings

frontend/src/
  pages/        ← vistas completas por ruta
  components/   ← componentes reutilizables agrupados por dominio
  hooks/        ← custom hooks (lógica de datos, WebSocket, SSO)
  store/        ← Zustand stores (auth, proyectos, eventos)
  lib/          ← api client (axios), roles helpers, constantes, utils
  types/        ← interfaces TypeScript por dominio
```

## Principios no negociables

1. **Código limpio siempre**: legible, modular y reutilizable. Evitar archivos y funciones gigantes.
2. **Seguridad por defecto**: todo cambio debe reducir riesgo, nunca aumentarlo.
3. **Producción primero**: todo cambio debe quedar Docker-ready y compatible con despliegue real.
4. **No romper flujos existentes**: cambios incrementales, retrocompatibles y con riesgo controlado.
5. **Sin deuda oculta**: no dejar TODO críticos sin registrar en `docs`.

## Estilo de código limpio

- Preferir funciones cortas con una sola responsabilidad.
- Nombrar variables y funciones con semántica de negocio clara (ej. `canEscalar`, `require_admin`).
- Evitar lógica duplicada; extraer helpers/utilidades reutilizables.
- Evitar condicionales anidados complejos; usar early return cuando aplique.
- Mantener contratos de API estables; si se cambia contrato, documentar migración.
- Si una sección no es obvia, agregar comentario breve y útil (no redundante).

## Reglas de arquitectura y separación de capas

- `routes/`: solo transporte HTTP, validación de entrada/salida con schemas Pydantic. Nunca lógica de negocio directa.
- `services/`: toda la lógica de negocio. Los routers los llaman, no al revés.
- `models/`: definición de tablas SQLModel únicamente. Sin lógica de negocio.
- `schemas/`: contratos de entrada/salida separados por dominio.
- `core/dependencies.py`: guards de autenticación y roles (`get_current_user`, `require_admin`, `require_superadmin`, etc.). Siempre usar los guards existentes, no reimplementar verificación de roles en otro lugar.
- Frontend `hooks/`: toda la lógica de datos y llamadas API. Las páginas y componentes solo consumen hooks.
- Frontend `store/`: estado global solo cuando es necesario cross-component (auth, eventos WS). No usar Zustand para estado local de formularios.
- No mezclar lógica de negocio dentro de componentes UI.

## Sistema de roles — nunca romper

La jerarquía de roles es inmutable: `usuario(0) → coordinador(1) → admin(2) → superadmin(3)`.

- Backend: usar siempre `require_coordinador`, `require_admin`, `require_superadmin` de `core/dependencies.py`.
- Frontend: usar siempre los helpers de `lib/roles.ts` (`isAdmin`, `isSuperAdmin`, `canEscalar`, `canSuperaprobar`, etc.). Nunca comparar roles con strings hardcodeados directamente en componentes.
- Si se agrega un nuevo guard de rol, hacerlo en `core/dependencies.py` (backend) y en `lib/roles.ts` (frontend). Nunca inline.

## Flujo de estados de proyecto — respetar siempre

El ciclo de vida de un proyecto sigue estados estrictos. Ningún cambio debe saltarse pasos ni crear transiciones no autorizadas. Los permisos por estado ya están definidos en `lib/roles.ts` — extender ahí, nunca en componentes.

## Ciberseguridad obligatoria

- **Nunca** hardcodear `SECRET_KEY`, `WEBHOOK_SECRET`, `SUPERADMIN_PASSWORD` ni credenciales en código fuente. Siempre por variables de entorno (`.env` + docker-compose).
- `SECRET_KEY` y `WEBHOOK_SECRET` en producción deben ser valores fuertes generados, nunca `CHANGE_ME`.
- Mantener CORS estricto: en producción `CORS_ORIGINS` debe listar dominios exactos, nunca `*`.
- Validación de entrada siempre en backend (tipos, formatos, longitudes, permisos). No confiar solo en validación frontend.
- No exponer información sensible en logs, errores HTTP ni respuestas de debug.
- El interceptor de Axios en `lib/api.ts` maneja expiración de token — no duplicar esta lógica.
- El `alert()` de sesión expirada en `api.ts` es deuda conocida — al modificar ese archivo, migrar a `sonner` toast.
- Proteger endpoints sensibles con las dependencias de roles correctas en cada router.
- Seguir Defense in Depth: red (Nginx/firewall), app (rate limit, validación, lockout), contenedor (sin privilegios innecesarios).

## Reglas Docker y despliegue

- Todo cambio debe funcionar dentro de los contenedores `matrix_backend` y `matrix_frontend`, no depender de binarios locales.
- `docker-compose.yml` es la fuente de verdad de ejecución.
- El backend expone solo el puerto `8000` internamente (`expose`, no `ports`). El frontend expone el `80`. No agregar puertos innecesarios al host.
- Variables de entorno sensibles van siempre en `.env` referenciadas desde `docker-compose.yml` con `${VAR}`.
- Si se agregan dependencias Python, verificar que el `Dockerfile` del backend las instale correctamente (`requirements.txt`).
- Si se agregan dependencias npm, verificar que el build del frontend siga funcionando en Docker.
- El volumen `sqlite_data` persiste la base de datos en `/app/data`. No cambiar esa ruta sin migrar el volumen.
- El `healthcheck` del backend usa `/health` — no eliminar ni cambiar ese endpoint.

## Base de datos y migraciones (SQLite + SQLModel)

- El proyecto usa SQLite con migraciones manuales via `run_migrations()` en `database.py`. No introducir Alembic sin consenso.
- Cualquier cambio de modelo debe ser backward compatible o incluir la migración correspondiente en `run_migrations()`.
- Operaciones destructivas deben ser transaccionales.
- `User`, `Project` y tablas core (roles, permisos) nunca deben modificarse sin plan de migración explícito.
- Optimizar primero con índices SQLModel (`index=True`) antes de introducir complejidad extra.
- Evitar N+1 queries en endpoints de alto uso — usar joins o eager loading cuando aplique.
- Los seeds (`admin_seed.py`, `questions_seed.py`) son idempotentes — mantener esa propiedad en cualquier modificación.

## Reglas para frontend

- **Nunca usar `alert()` bloqueantes** para errores de flujo. Usar `sonner` (ya instalado) para toasts.
- Usar estados de error/success visibles en UI con feedback claro al usuario.
- Evitar elementos hardcodeados cuando existe fuente dinámica desde backend.
- Mantener accesibilidad básica: labels en inputs, estados de loading, feedback visible en acciones async.
- Componentes enfocados y reutilizables. Si un componente supera ~250 líneas, considerar dividirlo.
- Animaciones con `framer-motion` (ya instalado) — no introducir otras librerías de animación.
- Gráficas con `recharts` (ya instalado) — no introducir otras librerías de charts.
- Iconos con `lucide-react` (ya instalado) — no introducir otras librerías de iconos.
- UI base con `shadcn/ui` + `radix-ui` (ya instalados) — no introducir otras librerías de componentes.

## WebSocket

- El manager de conexiones vive en `core/ws_manager.py`. Toda emisión de eventos WS pasa por ahí.
- Los stores de eventos (`commentEventStore`, `roiEventStore`) son los consumidores en frontend.
- El hook `useWebSocket.ts` maneja reconexión — no duplicar esa lógica.
- No emitir datos sensibles por WebSocket sin validar permisos del receptor.

## Observabilidad y auditoría

- Loggear eventos de seguridad y accesos denegados en módulos sensibles.
- Mensajes de log claros y accionables.
- No incluir secretos ni payloads sensibles en logs.
- Si un error es esperado, registrar `warning`; si compromete operación, registrar `error`.

## Flujo de trabajo para cambios

**Antes de codificar:**

- Revisar impacto en backend, frontend, seguridad y despliegue.
- Confirmar si afecta contratos API (`schemas/`), modelos o variables de entorno.
- Verificar si el cambio afecta el flujo de estados de proyecto o la jerarquía de roles.

**Durante implementación:**

- Hacer cambios pequeños y verificables.
- Priorizar claridad sobre "magia" o sobreingeniería.
- Respetar la separación de capas — no mezclar responsabilidades.

**Antes de cerrar:**

- Verificar build/test mínimo del módulo afectado.
- Verificar que sigue desplegable en Docker (`docker compose up --build`).
- Verificar que no se introdujeron secretos ni configuraciones inseguras.
- Actualizar `docs/` cuando cambie comportamiento, seguridad u operación.

## Definición de Done (obligatoria)

Un cambio se considera listo solo si:

- Compila/build pasa en el módulo afectado.
- No rompe endpoints/rutas existentes sin justificación documentada.
- Cumple criterios de seguridad (sin secretos hardcodeados, CORS correcto, roles validados).
- Es desplegable en Docker sin pasos ocultos.
- Queda documentado cuando aplique.

## Antipatrones prohibidos

- Hardcodear `SECRET_KEY`, passwords, tokens o datos corporativos sensibles.
- Comparar roles con strings inline en componentes — usar siempre `lib/roles.ts`.
- Lógica de negocio directamente en routers o componentes UI.
- Funciones/archivos gigantes sin separación por responsabilidades.
- Soluciones "temporales" sin trazabilidad ni plan de retiro.
- Cambios que solo funcionan en entorno local (dependencias de binarios locales, rutas absolutas del dev, etc.).
- Bypass de validaciones, permisos o controles de seguridad.
- Introducir librerías que dupliquen funcionalidad ya cubierta por el stack existente.
- Modificar `run_migrations()` sin probar el upgrade desde una BD existente.

## Regla de continuidad

Si existe conflicto entre velocidad y calidad:

- Gana seguridad y estabilidad.
- Luego se optimiza velocidad de entrega.
